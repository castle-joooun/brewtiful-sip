# brewtiful-sip 기능명세서 (Phase 1 — MVP)

버전: v0.3 (원두 동적 API화 반영 — ADR-0002)
관련 문서: `CLAUDE.md`, `docs/ERD.md`(예정), `docs/api-docs/`(예정)

---

## 1. 개요

홈카페 방문 손님이 QR코드로 웹사이트에 접속해 원두를 확인하고 메뉴를 주문하며, 운영자는 주문을
접수·처리하고, 손님은 자신이 실제로 주문한 건에 한해서만 리뷰를 남길 수 있는 서비스.

## 2. 액터(Actor)

| 액터 | 설명 |
|---|---|
| 손님(Guest) | 로그인 없이 QR로 접속. 별도 회원가입 없음 |
| 운영자(Owner) | 매장 관리자. 원두/메뉴 관리, 주문 접수 및 준비완료 처리 |

## 3. 기능 목록

### 3.1 QR 접속
- 매장에 비치된 QR코드를 스캔하면 웹사이트 메인(원두 리스트 또는 메뉴 화면)으로 이동한다.
- **[확정]** QR은 매장 단위 고정 URL 1개. 테이블 식별자는 도입하지 않는다.
- QR 자체는 단순 정적 URL이며 별도 인증 토큰을 포함하지 않는다 (인증은 주문 이후 시점에 발급).

### 3.2 원두 리스트 조회
- **[확정 — ADR-0002, v0.3에서 변경]** Phase 1에서 원두 목록을 **DB에서 관리하고 조회 API
  (`GET /beans`)로 동적 노출**한다. 프론트는 하드코딩이 아니라 API를 호출해 원두 정보를 로딩한다.
  (기존 v0.2의 "프론트 정적 HTML 하드코딩" 방침을 폐기)
- 원두 상세 정보(이름, 원산지, 로스팅 포인트, 가공 방식, 향미 노트)는 `bean` 테이블에서 관리한다.
  품절/노출 제어는 `active` 플래그로 처리하고, 조회 API는 `active=true`만 노출한다. 수량 기반
  재고 관리는 Phase 1 범위에서 제외(필요 시 Phase 2에서 재검토).
- **원두 등록/수정(쓰기)**은 Phase 1에서 관리자 API 없이 시드 SQL(`docs/sql/`)로 관리한다
  (메뉴와 동일 방침, 3.7 참고). 조회 API만 구현한다.
- **[확정 — ADR-0001]** 브루잉 커피처럼 원두 선택이 필요한 메뉴 주문 시, 주문 항목(`OrderItem`)은
  **`Bean` 엔티티를 `bean_id` FK로 참조**한다. 원두가 DB 단일 출처(ADR-0002)이므로 초기에
  우려했던 이중 관리 없이 참조 무결성을 확보한다. (근거/트레이드오프는 `docs/decisions.md`
  ADR-0001·0002, 스키마는 `docs/ERD.md` 참고)

### 3.3 메뉴 선택 및 주문 생성
- 메뉴 목록(예: 아이스 아메리카노, 브루잉 커피 등) 중 선택하여 주문을 생성한다.
- 브루잉 커피처럼 원두 선택이 필요한 메뉴는 3.2의 원두 중에서 선택 가능해야 한다 (품절 원두 제외).
- 주문 생성 시 서버는 **주문 토큰(orderToken, UUID)** 을 발급하고, 손님은 이 토큰이 포함된
  URL(예: `/orders/{orderId}?token=xxxx`)로 주문 상태를 조회한다.
- 주문 상태(OrderStatus): `RECEIVED`(접수) → `PREPARING`(제조중, 선택적) → `READY`(준비완료)
- **[확정]** 한 번의 주문에 여러 메뉴 항목을 담을 수 있다 (장바구니 개념 지원). 주문(Order)은
  1개 이상의 주문 항목(OrderItem)을 가지는 구조로 설계한다.

### 3.4 주문 알림 (운영자용)
- 새 주문이 생성되면 운영자에게 알림이 전달되어야 한다.
- **[확정] Phase 1**: 카카오톡 등 외부 연동 없이, 서버 콘솔 로그 + 운영자 전용 대시보드로
  전달한다. 대시보드는 **SSE(Server-Sent Events) 또는 WebSocket 기반 서버 → 대시보드 실시간
  push**로 구현한다 (외부 서비스 연동이 아니므로 Phase 1 범위 내).
- **Phase 2**: 실제 카카오톡 알림톡 또는 대체 수단(예: 슬랙 웹훅) 연동 — 로드맵상 이번 단계에서는
  진행하지 않는다.

### 3.5 주문 준비완료 처리
- 운영자가 특정 주문을 "준비완료(READY)"로 상태 변경한다.
- **[확정 — ADR-0003]** 운영자 인증은 회원 계정이 아니라 **고정 마스터 코드(master_code)**를
  `X-Master-Code` 헤더로 전송해 검증한다. 코드 값은 하드코딩하지 않고 설정/환경변수로 외부화하며,
  서버에서 상수시간 비교한다. (Phase 2에서 정식 인증으로 승격 — decisions.md 참고)
- 상태 변경 시점을 기록한다 (리뷰 작성 가능 기간 계산에 사용, 3.6 참고).
- **[확정]** 손님 화면도 3.4와 동일하게 SSE/WebSocket push로 상태 변경을 실시간 반영한다.

### 3.6 리뷰 작성 (제약 포함)

**제약 정책 (확정):** 주문 토큰(orderToken) 기반 인증 + **메뉴(주문 항목) 단위** 리뷰.

- 리뷰 작성 화면은 로그인이 아니라, 주문 생성 시 발급된 `orderToken`을 가진 사람만 접근 가능하다.
  (주문 완료/준비완료 화면에서 "리뷰 남기기" 버튼 → 토큰을 `X-Order-Token` 헤더로 전달)
  손님 주문상태 조회 GET은 공유 링크 설계(3.3)를 위해 예외적으로 token을 쿼리 파라미터로 유지한다.
- **리뷰 단위 = 메뉴(OrderItem)**: 리뷰는 주문 전체가 아니라 **주문에 담긴 개별 메뉴 항목**에
  대해 작성한다. 한 주문에 메뉴가 여러 개 담겨 있으면 항목별로 각각 리뷰를 남길 수 있다.
- **1 주문 항목(OrderItem)당 1리뷰**: 이미 리뷰가 작성된 주문 항목은 재작성 불가
  (`review 존재 여부로 체크` 또는 `used` 플래그).
- **작성 가능 시점**: 주문 상태가 `READY`(준비완료) 이후에만 리뷰 작성 가능 (준비 전 리뷰 방지).
- **작성 가능 기간**: 준비완료(READY) 시점으로부터 **3일 이내**로 제한한다.
- 리뷰 내용: 평점(1~5), 텍스트 후기(선택), 작성일시, 대상 주문 항목(메뉴) 참조.
- 리뷰는 별도 승인 없이 즉시 공개 노출한다 (Phase 1). 운영자 승인제(비공개 → 승인 후 공개)는
  악성 리뷰 이슈 발생 시 Phase 2에서 검토.
- 리뷰 목록 조회는 손님 누구나 볼 수 있다 (메인 화면 또는 별도 리뷰 페이지). 메뉴 단위이므로
  "메뉴별 리뷰 모아보기" 형태의 조회도 가능하다.

### 3.7 (참고) DB 관리 방식
- Phase 1은 별도 관리자 UI 없이, 메뉴 등록·수정을 SQL 쿼리 또는 최소한의 관리자 API로 수행한다
  (원두는 3.2 확정에 따라 DB 관리 대상이 아니라 프론트엔드 정적 콘텐츠). `docs/sql/` 폴더에
  초기 데이터 및 자주 쓰는 쿼리를 메모장 형태로 관리한다.
