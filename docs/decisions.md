브루틀리적인 설계 결정을 짧게 기록한다. 형식: 컨텍스트 → 결정 → 트레이드오프 → 상태.

## ADR-0001: 원두 참조는 Bean 엔티티 + FK

- 날짜: 2026-07-04
- 상태: 채택(Accepted). 초기 우려였던 이중 관리 단점은 ADR-0002로 해소됨.
- 관련: `docs/기능명세서.md` 3.2, `docs/ERD.md`, [ADR-0002](#adr-0002-원두-조회를-프론트-하드코딩에서-db--조회-api로-전환)

### 컨텍스트
명세 3.2에서 Phase 1 원두 정보는 프론트엔드 정적 콘텐츠(하드코딩)로 관리하기로 확정했다.
그러나 브루잉 커피처럼 원두 선택이 필요한 메뉴 주문 시, `OrderItem`이 원두를 어떻게 가리킬지가
미결로 남아 있었다. 선택지는 (a) 원두명 문자열 스냅샷 저장, (b) 최소 `Bean` 엔티티 + FK,
(c) 애플리케이션 Enum/코드 참조.

### 결정
(b) **최소 `Bean` 엔티티(id, name, active)를 두고 `OrderItem.bean_id` FK로 참조**한다.
원두 상세(원산지, 로스팅, 향미 등)는 여전히 프론트 정적 콘텐츠로 유지하고, DB `bean`
테이블은 참조 무결성 확보에 필요한 최소 컬럼만 보유한다.

### 트레이드오프
- 장점: 주문–원두 참조 무결성을 DB 레벨에서 보장. 원두별 통계/조회 확장 여지. 정규화 관점에서
  면접 설명력이 좋음.
- 단점: "프론트 정적 목록 ↔ `bean` 테이블" **이중 관리** 발생. → 초기 시드 SQL(`docs/sql/`)로
  동기화하고, 실시간 품절 등 요구가 생기면 Phase 2에서 원두 API화를 재검토한다.
- 반대 관점(문자열 스냅샷 (a))도 유효했다: 테이블/조인 없이 단순하고, 명세의 "원두=정적 콘텐츠"
  기조와 더 일치하며, MSA 분리 시 order 도메인이 bean에 결합되지 않는다. 무결성보다 단순성을
  우선했다면 (a)가 나았을 수 있어, 이중 관리 부담이 커지면 재검토 대상이다.

### 후속
- Phase 3 MSA 분리 시, 경계를 넘는 `bean_id` FK는 물리 제약을 풀고 논리 참조로 전환 검토
  (`docs/ERD.md` 6장).

## ADR-0002: 원두 조회를 프론트 하드코딩에서 DB + 조회 API로 전환

- 날짜: 2026-07-04
- 상태: 채택(Accepted). 기능명세서 v0.2의 "프론트 정적 HTML 하드코딩" 방침을 대체(Supersedes).
- 관련: `docs/기능명세서.md` 3.2, `docs/ERD.md`, [ADR-0001](#adr-0001-원두-참조는-bean-엔티티--fk)

### 컨텍스트
초기 명세(3.2)는 Phase 1 원두 정보를 프론트엔드 정적 HTML로 하드코딩하기로 했다. 그러나
ADR-0001로 `bean` 테이블이 도입되면서 "프론트 정적 목록 ↔ `bean` 테이블"을 손으로 맞춰야 하는
이중 관리가 생겼다. 원두 하나 추가 시 프론트와 DB 양쪽을 갱신해야 하고, 하나라도 누락되면
브루잉 주문의 `bean_id` FK 참조가 깨진다.

### 결정
원두 상세를 `bean` 테이블에서 관리하는 **단일 출처(single source of truth)**로 두고,
`GET /beans` 조회 API로 노출한다. 프론트는 하드코딩 대신 API를 호출해 동적으로 원두를 로딩한다.
- 품절/노출 제어는 `bean.active` 플래그 하나로 처리하고, 조회 API는 `active=true`만 반환한다.
  수량 기반 재고(`stock_quantity`)는 동시성·차감 로직 부담이 있어 Phase 1 범위에서 제외한다.
- 원두 등록/수정(쓰기)은 Phase 1에서 관리자 API 없이 시드 SQL(`docs/sql/`)로 관리한다
  (메뉴와 동일 방침). 조회 API만 구현한다.

### 트레이드오프
- 장점: 이중 관리 소멸(원두의 단일 출처가 DB). ADR-0001 FK의 무결성 단점 해소. 이후 품절 실시간
  반영·원두별 통계 등 확장에 자연스럽게 연결. 프론트/백엔드 책임 분리가 깔끔해짐.
- 단점: Phase 1 스코프가 소폭 증가(원두 조회 API 1개 + Bean 상세 엔티티). 단, Redis/Kafka 등
  확장 단계 기술을 끌어오는 것은 아니며, "원두 리스트 조회"는 CLAUDE.md Phase 1 기능 목록에
  이미 포함돼 로드맵 위반이 아니다(구현 방식만 변경).
- 반대 관점: 트래픽이 사실상 없는 홈카페에서 정적 하드코딩이 가장 단순하다는 주장도 유효하다.
  다만 이력서/포트폴리오 목적상 API·계층 분리를 보여주는 편이 설명력이 크고, 이중 관리 제거라는
  실익도 있어 API화를 택했다.

### 후속
- `docs/api-docs/`에 `GET /beans` OpenAPI 초안 작성.
- 초기 원두 시드 SQL을 `docs/sql/`에 정리.

## ADR-0003: 운영자 인증은 고정 마스터 코드(헤더)

- 날짜: 2026-07-04
- 상태: 채택(Accepted)
- 관련: `docs/기능명세서.md` 3.5, `docs/api-docs/openapi.yaml`

### 컨텍스트
Phase 1에는 운영자 계정/회원 개념이 없다. 운영자 전용 동작(주문 상태 변경, 대시보드)만
보호하면 되며, 로그인·세션·권한 모델을 도입하는 것은 MVP 대비 과설계다.

### 결정
운영자 전용 엔드포인트(`PATCH /orders/{orderId}/status` 등)는 **고정 마스터 코드**를
`X-Master-Code` 헤더로 받아 검증한다.
- 코드 값은 소스에 하드코딩하지 않고 **설정/환경변수(`application.yml` + env)로 외부화**한다.
- 비교는 타이밍 공격을 피하도록 **상수시간 비교**로 한다.
- 검증 실패 시 401을 반환한다. 인증 처리는 Controller가 아니라 필터/인터셉터(또는 Spring
  Security 최소 설정) 계층에 두어 Controller를 얇게 유지한다.

### 트레이드오프
- 장점: 계정 관리 없이 즉시 도입 가능, MVP 스코프에 부합, Controller 관심사 분리 유지.
- 단점: 공유 정적 비밀 → 회전/폐기·감사 로그·다중 운영자 구분 불가, 유출 시 전면 노출.
- 완화: HTTPS 전제, 값 외부화, 상수시간 비교. 로그에 코드 값 미기록.
- 반대 관점: 처음부터 Spring Security 기반 운영자 로그인/JWT가 "정석"이나, 유저스토리(운영자
  1인) 대비 과설계라 보류했다.

### 후속
- Phase 2에서 운영자 계정 + 정식 인증(Spring Security, 세션/JWT)으로 승격 검토.
- 손님 리뷰 작성 토큰은 헤더(`X-Order-Token`)로 전송한다.
- [확정] 손님 주문상태 조회 GET은 공유 가능한 딥링크(명세 3.3) 특성상 token을 쿼리 파라미터로
  유지한다. URL 토큰은 접근 로그/브라우저 이력/Referer로 노출될 수 있으나, 대상이 저위험(주문
  상태 조회)이고 링크 공유 UX가 핵심이라 수용한다. 완화: HTTPS 전제, 액세스 로그의 token 쿼리
  마스킹, 토큰은 주문별 1회성 UUID.
